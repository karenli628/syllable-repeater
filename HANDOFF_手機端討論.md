## 1. 討論起點：PLAN3.0 的商用缺口

PLAN3.0 設計是「桌面製作課件、手機只練習」。缺口：
**只有手機、沒有桌機的陌生使用者，下載 App 後是空的**——他沒有 `.abopack`，也無法自己做。
個人自用不受影響；但這是商用的真實障礙。

## 2. 用戶的商用想法（已提出）

手機版除了「匯入課件檔（.abopack）」，**再增加「匯入 mp3/音檔，手機自己辨識製作」**。
由此拆成**三套方案**：

| 方案 | 適合對象 | 辨識在哪做 | PLAN3.0 現況 |
|---|---|---|---|
| 桌面 + 手機 | 自用 / 認真使用者 | 桌面（whisper.cpp + demucs.cpp） | ✅ 已涵蓋 |
| 純桌面 | 只在電腦練 | 桌面 | ✅ 已涵蓋 |
| **純手機** | 只有手機的陌生使用者（商用關鍵） | **手機端（須輕量，難點）** | ❌ 待設計 |

用戶原想參考 https://nackpan.net/blog/ 作者的辨識技術。

## 3. 關鍵情報調查結果（已完成）

調查 nackpan.net 作者的 Repete Plus（語言學習播放器）：

- 核心舊功能：**無音分割**（靜音偵測自動分段）— 純 DSP，極輕量，純手機可跑
- v11（2026-05-07）新增「音声認識 + 発声正確度比較」，**月額 200 円訂閱**
- v11.2（2026-05-15）新增「文字起こし」（轉錄）

**決定性線索（強推論，非官方確認）：**
v11 文章原文：「発声チェック関連の音声認識機能は、**iOS 26以降の対応機種**で利用できます。」

→ 若用 Whisper（自帶模型）**不會綁 OS 版本**。會綁「iOS 26+ 對應機種」的，
**只有 OS 內建 API** → 幾乎可斷定他用 **Apple iOS 26 原生 SpeechAnalyzer / SpeechTranscriber**
（2025 WWDC 發表的新一代 on-device ASR，取代舊 SFSpeechRecognizer）。
作者對技術刻意保持沉默，三篇文章都沒明寫，但證據強烈。

## 4. 技術判斷：靜音分段 vs 原生 ASR

| | 靜音分段（nackpan 舊技術） | 平台原生 ASR |
|---|---|---|
| 切出單位 | 語句/呼吸段（停頓之間） | word-level → 配 CMUdict 可切**音節** |
| 能做「逐音節疊加」嗎 | ❌ 只能語句級疊加 | ✅（iOS 可，Android 不一定） |
| 重量 | 極輕，與 PLAN3.0 §3.2 自寫停頓偵測同技術 | 零模型體積（OS 提供） |
| 原聲不可替換(§0.1) | ✅ | ✅ |

**重要**：若純手機只用靜音分段，招牌功能「communication → com·mu·ni·ca·tion 逐音節疊加」**做不到**，手機只能「整句之間」疊加。原生 ASR 才能在手機上保住音節級。

## 5. iOS / Android 各用原生 ASR 的優劣（已分析）

**共通優勢**：零模型體積、離線免費無 token、省電、維護成本低。

**關鍵劣勢 — 兩端能力不對等：**

| | iOS（SpeechAnalyzer/Transcriber） | Android（SpeechRecognizer） |
|---|---|---|
| 設計用途 | 新、強，支援檔案轉錄 + word timing | 老、弱，為「語音指令」設計，非長音檔轉錄 |
| word timing | 較可靠 → 能切音節 | 常給不出可靠 timing → **切不出音節** |
| 版本/碎片化 | 綁 iOS 26+，但統一 | 綁版本 + **各廠牌引擎不同，品質不一** |
| 長音檔離線 | 較好 | 普遍差，常需切段或連網 |

**一句話結論**：iOS 原生夠好可撐「純手機逐音節疊加」；
Android 原生對長音檔/word timing 普遍不可靠，撐不起同等功能 → **雙系統真實裂縫**。
