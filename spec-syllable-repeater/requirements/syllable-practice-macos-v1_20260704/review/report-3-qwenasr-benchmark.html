報告三：QwenASRMiniTool 借鏡分析

對照專案：github.com/karenli628/QwenASRMiniTool（Windows／Python／已出貨，CTk 1.0.9＋WebView 0.2 雙版本線）｜ 本專案：Syllable Repeater macOS v1（Dart/Flutter，93%）｜ 分析日期：2026-07-07（shallow clone 實讀原始碼）
一句話結論：最值得借的是它「重工件按需下載＋工具就緒狀態 UI＋腳本化發布」這一整套出貨經驗——正好對準本專案僅剩的高風險段（2.1 sidecar 實體與 9.1/9.2 發布）；同時有 4 個地雷絕不可照抄（GPL FFmpeg 下載源、SSL 不驗證降級、3300 行單檔 UI、伺服器/批次功能），照抄任一個都會直接違反本專案的 M9／安全基線／架構守則／Non-scope。

2. 可借鏡清單（依「加速完成、降低失敗率」排序）

優先 1 工件下載器模式 → 直接解 2.1 的「等工件」僵局

出處downloader.py（840 行，純標準庫）：斷點續傳、SHA 完整性驗證、主倉庫失敗自動切備援倉庫、鏡像站改寫、下載進度回呼。

落點寫一支 scripts/fetch_sidecar_artifacts.py（或並入 prepare_release_sidecars.py 前段）：從妳核可的來源下載 LGPL-shared FFmpeg/ffprobe、whisper.cpp 二進位、ggml-small.en.bin、demucs.cpp 二進位＋ggml-model-htdemucs-4s-f16.bin，每個工件寫死 SHA-256＋來源 URL＋授權 三元組，下載完自動接跑既有 license gate 與 staging。

效益2.1 從「手動蒐集工件、換機重來」變成「一條指令可重現」；同時是 9.1 CI 化與未來 Apple Silicon 版的基礎。注意：demucs.cpp 二進位大概率仍需本機自行編譯（上游未必發布 x86_64 release）——腳本設計成「有 URL 就下載、沒 URL 就檢查本機產物並提示編譯指令」。

優先 2 「工具就緒狀態」UI 模式 → 把降級從隱性變顯性

出處QwenASR 開機不強制下載模型：模型頁顯示每個核心「已就緒／未下載」＋「下載並載入」按鈕，其餘功能照常可用。

落點本專案已有半成品（demucsReadyProvider＋未就緒 tooltip、missingPaths() 缺件降級 preview runner）——建議在設定頁加一個「分析工具狀態」區塊：逐列 ffmpeg／whisper／模型／cmudict／demucs 的就緒狀態與路徑，缺件時顯示取得指引（未來可接優先 1 的下載器）。

效益把「靜默 fallback 到 preview runner」（審查發現 S-001 的隱性行為）變成使用者看得懂的狀態，換機/親友試用時自助排錯，支援成本大降。

優先 3 腳本化發布 → 9.2 的樣板

出處make_release_zip.bat＋version.py：版本號單一事實來源、staging 排除大型工件與個人資料（robocopy /XD）、產出版本化 ZIP、末尾直接印「下一步：開 Release、上傳資產」。

落點寫 scripts/make_release_zip.py（Python，與現有 scripts 一致）：讀取單一版本常數 → 打包未簽章 .app → 排除 dev 工件 → 產 SHA256SUMS → 隨附 安裝說明.md（Gatekeeper 略過步驟）→ 印出 GitHub Release 待辦三行。9.2 的驗收（AT-09-03）就以這個腳本產物為準。

效益發布從「一次性手工」變成可重複、可寫進 release-checklist.md 的固定程序。

優先 4 README／安裝說明的寫法 → 9.2 文件範式

出處QwenASR README：第一段就是價值主張＋隱私承諾（「資料不離開你的電腦」）、每個分頁一張截圖、目標讀者設定為「同事、同學、阿嬤」、版本線清楚、舊文件另檔隔離（oldreadme.md）。

落點本專案 repo 目前沒有使用者導向 README。9.2 時比照結構寫：①這是什麼＋M10 隱私承諾（錄音用完即刪，是天然賣點）②安裝三步（下載 zip → xattr -cr 或右鍵開啟 → 完成）附截圖 ③功能導覽每頁一圖 ④已知限制（Intel 優先、10 分鐘上限）。開發文件（PLAN3.0、交接檔）另檔隔離，README 只服務使用者。

優先 5 模型檔預檢（--show 驗證）→ 消滅「壞模型神祕失敗」

出處fa_aligner.py：載入前先跑 main.exe --show 確認模型檔真的是 ForcedAligner，不對就降級並明講原因；失敗一律回 [] 讓呼叫端退回比例估算。

落點missingPaths() 目前只檢查「檔案存在」——可加輕量預檢：模型檔大小下限＋magic bytes（ggml/gguf 頭），或首次啟動用 1 秒靜音 WAV 跑一次 whisper 煙霧測試並快取結果。下載中斷的半個模型檔（466MB 的 small.en 很容易發生）就不會以「辨識神祕失敗」呈現。

附註它「FA 失敗→退回比例估算」與本專案「等比例切＋needsReview」是同一個設計哲學——互相印證這條路是對的。它用的 qwen3-forcedaligner（GGUF）也是未來「切點精度外掛」候選之一（與 MFA 並列，屆時先過 M9 授權核對）。

優先 6 孤兒行程回收 → M4 的最後一塊補強

出處proc_guard.py：Windows Job Object KILL_ON_JOB_CLOSE——主行程死（含強殺/崩潰），OS 自動連坐回收所有子行程；模組級持有 handle 防 GC 提早觸發；非 Windows 靜默降級。

落點macOS 上父行程死亡不會自動殺子行程：使用者在 whisper 跑到一半強退 App，whisper-cli 會變孤兒繼續燒 CPU。SidecarRunner 已處理 timeout/kill，但管不到「App 本體被強殺」。低成本補法：App 啟動時清掃（沿用「啟動 clearTemp」哲學）——sidecar 啟動時把 PID 記到 temp 下的 pidfile，App 下次啟動比對仍存活的舊 PID（驗證行程名）後回收。Windows 版（Phase 2）直接翻譯 proc_guard 的 Job Object 作法。

效益單人自用影響小，屬 M4 完整性的收尾；建議記入 backlog 而非 v1 必做（守 MVP 邊界）。

優先 7 GitHub Releases 自我更新 → Phase 2 роadmap

出處updater.py：查 Releases API 比對版本 → 下載 ZIP → helper script 等主程式退出後疊加覆蓋（保留使用者資料目錄）→ 重啟。更新包刻意不含模型（小包快傳）。

落點v1 不做（Non-scope 紀律；免簽章 App 自我更新在 macOS 上還會反覆觸發 Gatekeeper，體驗差）。記入 Phase 2／取得 Developer ID 之後再評估；屆時「更新包不含 sidecar 模型」的拆分策略直接沿用。

3. 絕不可照抄清單（照抄即違規）

  1. GPL FFmpeg 下載源。QwenASR 下載的是 BtbN …-win64-gpl-essentials.zip——對它（免費工具、不在意 GPL）沒事，對本專案是 M9 紅線直接違反。借下載器「模式」時，來源必須換成 LGPL-shared build 並過既有 license gate（gate 已會擋 --enable-gpl，不要繞）。
  2. SSL 驗證降級。downloader._ssl_ctx() 最後一層 fallback 是 CERT_NONE（不驗證憑證）。本專案的安全基線（matrix #20 https-only、#31 host allowlist 的精神）不允許——寫下載器時憑證驗證失敗就失敗，改靠 SHA-256 pinning 保完整性，不做不驗證降級。
  3. 單檔巨型 UI。app.py 3304 行是它的歷史債（後期才拆 WebView 後端）。本專案「一 screen 一檔＋controller 分離」是刻意設計，任何「先塞同一檔再說」都是倒退。
  4. 伺服器／通道／批次功能。api_server.py、cf_tunnel.py、批次分頁對它是賣點，對本專案是 Non-scope 5（無伺服器）與 v1 不做批次的明確範圍外——看到別人有，不等於要加（憲法 C5）。

5. 順帶的反向印證（本專案已經做對的事）

  • QwenASR 的「失敗退回比例估算」＝本專案「等比例切＋needsReview」——同一哲學，已內建。
  • QwenASR 用「更新包不含模型」控制體積＝本專案「MFA/CREPE/WORLD 外掛化、零 Python」的既定決策——方向一致。
